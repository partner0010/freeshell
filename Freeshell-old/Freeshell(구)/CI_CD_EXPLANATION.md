# CI/CD 구축 설명서

## 🤔 CI/CD가 뭔가요?

### CI/CD란?

**CI/CD**는 **Continuous Integration (지속적 통합)**과 **Continuous Deployment (지속적 배포)**의 약자입니다.

쉽게 말하면:
- **코드를 작성하면 자동으로 테스트하고 배포하는 시스템**

---

## 📝 상세 설명

### 1. CI (Continuous Integration) - 지속적 통합

**무엇을 하나요?**
- 코드를 GitHub에 올리면 (push)
- 자동으로 테스트 실행
- 자동으로 코드 품질 검사
- 자동으로 빌드 (프로그램 컴파일)

**예시**:
```
1. 개발자가 코드 작성
2. GitHub에 업로드 (git push)
3. 자동으로:
   - 테스트 실행 ✅
   - 코드 오류 검사 ✅
   - 프로그램 빌드 ✅
4. 문제 있으면 알림 📧
```

**장점**:
- ✅ 버그를 빨리 발견
- ✅ 코드 품질 유지
- ✅ 수동 작업 불필요

---

### 2. CD (Continuous Deployment) - 지속적 배포

**무엇을 하나요?**
- 테스트 통과하면
- 자동으로 서버에 배포
- 자동으로 재시작

**예시**:
```
1. 테스트 통과 ✅
2. 자동으로:
   - 서버에 코드 배포 ✅
   - 프로그램 재시작 ✅
   - 배포 완료 알림 📧
```

**장점**:
- ✅ 수동 배포 불필요
- ✅ 빠른 업데이트
- ✅ 실수 방지

---

## 🎯 우리 프로그램에 CI/CD가 필요한 이유

### 현재 상황 (CI/CD 없음)

**수동 작업**:
```
1. 코드 작성
2. 로컬에서 테스트 (수동)
3. 서버에 업로드 (수동)
4. 서버에서 빌드 (수동)
5. 서버 재시작 (수동)
6. 문제 있으면 다시 1번부터...
```

**문제점**:
- ❌ 시간 소모
- ❌ 실수 가능성
- ❌ 배포 과정 복잡

---

### CI/CD 구축 후

**자동 작업**:
```
1. 코드 작성
2. GitHub에 push
3. 자동으로:
   - 테스트 실행 ✅
   - 빌드 ✅
   - 서버 배포 ✅
   - 재시작 ✅
4. 완료! 🎉
```

**장점**:
- ✅ 시간 절약
- ✅ 실수 방지
- ✅ 빠른 배포

---

## 🛠️ CI/CD 구축 방법

### GitHub Actions 사용 (추천)

**설정 파일**: `.github/workflows/deploy.yml`

```yaml
name: Deploy

# GitHub에 push하면 자동 실행
on:
  push:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: 테스트 실행
        run: npm test
  
  build:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: 프로그램 빌드
        run: npm run build
  
  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - name: 서버에 배포
        run: |
          # 서버에 코드 업로드
          # 서버에서 재시작
```

## 📊 CI/CD 구축 전후 비교

### 구축 전
- 배포 시간: **30분 ~ 1시간**
- 실수 가능성: **높음**
- 배포 빈도: **주 1~2회**

### 구축 후
- 배포 시간: **5~10분** (자동)
- 실수 가능성: **낮음**
- 배포 빈도: **매일 가능**

---

## ✅ 결론

**CI/CD = 자동화된 배포 시스템**

**필요한 이유**:
1. 시간 절약
2. 실수 방지
3. 빠른 업데이트
4. 코드 품질 유지

**우선순위**: 중간 (지금 당장 필수는 아님)
